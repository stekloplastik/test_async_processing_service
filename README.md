# Asynchronous Payment Processing Service

Асинхронный микросервис для обработки платежей с гарантией доставки сообщений (Transactional Outbox Pattern), встроенным механизмом идемпотентности, обработкой ошибок с повторными попытками (Exponential Backoff) и очередью недоставленных сообщений (DLQ).

Проект построен по принципам **Чистой Архитектуры (Clean Architecture)**, обеспечивая разделение бизнес-логики и инфраструктурных деталей.

---

## 🛠 Технологический стек

- **Язык**: Python 3.10+
- **Управление зависимостями**: Poetry
- **HTTP API**: FastAPI + Uvicorn
- **Брокер сообщений**: RabbitMQ + FastStream (с интеграцией Quorum Queues)
- **База данных**: PostgreSQL
- **ORM / Подключение к БД**: SQLAlchemy 2.0 (асинхронный драйвер `asyncpg`)
- **Миграции БД**: Alembic
- **Тестирование**: Pytest + HTTPX
- **Линтер и форматировщик**: Ruff
- **Контейнеризация**: Docker + Docker Compose

## 📐 Архитектура проекта (Clean Architecture)

Кодовая база разделена на три основных слоя:

1. **Domain Layer (Доменный слой)** (`app/domain/`):
   - Содержит чистые бизнес-сущности (`Payment`, `OutboxEvent`) и правила их валидации.
   - Не зависит от каких-либо внешних библиотек, ORM или веб-фреймворков.
   - Содержит доменные исключения.

2. **Use Cases Layer (Слой сценариев использования / Прикладной слой)** (`app/usecases/`):
   - Содержит логику бизнес-процессов: создание платежа, получение статуса платежа, обработка транзакции (имитация эквайринга), отправка вебхуков.
   - Определяет интерфейсы (порты) репозиториев (`PaymentRepository`, `OutboxRepository`), брокера событий (`EventPublisher`) и шлюза вебхуков (`WebhookGateway`).
   - Зависит исключительно от доменного слоя.

3. **Infrastructure & Adapters Layer (Инфраструктурный слой)** (`app/infrastructure/`):
   - Реализует интерфейсы из прикладного слоя (репозитории на SQLAlchemy, клиент вебхуков на HTTPX).
   - Содержит внешние фреймворки (FastAPI для роутинга, FastStream для консьюмеров RabbitMQ).
   - Содержит ORM-модели базы данных и конфигурацию Alembic.

```
app/
├── domain/                  # --- DOMAIN LAYER (Бизнес-правила) ---
│   ├── entities/            # Сущности (Payment, OutboxEvent)
│   └── exceptions.py        # Исключения
├── usecases/                # --- USE CASES LAYER (Прикладная логика) ---
│   ├── interfaces/          # Порты / Интерфейсы репозиториев и внешних сервисов
│   ├── create_payment.py    # Создание платежа + Идемпотентность + Запись в Outbox
│   ├── get_payment.py       # Получение данных платежа
│   ├── process_payment.py   # Эмуляция внешнего эквайринга
│   └── send_webhook.py      # Отправка вебхука с повторными попытками
└── infrastructure/          # --- INFRASTRUCTURE LAYER (Адаптеры и детали) ---
    ├── models/              # ORM-модели (SQLPayment, SQLOutbox)
    ├── repositories/        # Репозитории на SQLAlchemy 2.0 (SKIP LOCKED)
    ├── broker/              # Настройки FastStream (Quorum queues, DLQ, Exchange)
    ├── gateways/            # Шлюз вебхуков (HTTPX клиент)
    └── alembic/             # Миграции базы данных
```

## 🔑 Ключевые паттерны и технические решения

### 1. Транзакционный Outbox Pattern (Transactional Outbox)
Чтобы избежать распределенных транзакций и гарантировать доставку событий о новых платежах во внешние системы (или воркеры), используется паттерн Outbox:
- При создании платежа запись о платеже и событие `OutboxEvent` (`payments.new`) сохраняются в PostgreSQL в рамках **одной ACID-транзакции**.
- В фоне внутри процесса API работает фоновый поток **Outbox Relay**, который периодически опрашивает таблицу событий.
- Для исключения конкуренции и блокировок при масштабировании API-сервиса, опрос базы выполняется с использованием конструкции `SELECT ... FOR UPDATE SKIP LOCKED`.
- После успешной публикации события в RabbitMQ его статус в БД обновляется на `PROCESSED`. Это гарантирует доставку **At-Least-Once** (хотя бы один раз).

### 2. Идемпотентность API запросов
Сервис защищен от повторных случайных списаний/запросов со стороны клиентов при сбоях сети:
- Запрос на создание платежа требует заголовка `Idempotency-Key` (уникальный UUID).
- Если платеж с таким ключом уже существует в БД:
  - Сверяются параметры платежа (сумма, валюта, вебхук, метаданные). Если они идентичны — возвращается существующий платеж (статус `202 Accepted`).
  - Если параметры отличаются — возвращается ошибка `409 Conflict`.

### 3. Гарантии доставки и обработка ошибок (Retry Policy & DLQ)
- **Экспоненциальные повторы вебхуков**: При сбое отправки вебхука клиенту (например, если его сервер недоступен) Use Case выполняет 3 попытки отправки с экспоненциальной задержкой (`base_delay * (2 ** attempt)`).
- **Quorum Queues & Dead Letter Queue (DLQ)**:
  - Очередь `payments.new` настроена как **Quorum Queue** для обеспечения высокой доступности и репликации в кластере RabbitMQ.
  - Очередь связана с `payments.dlx` (Dead Letter Exchange).
  - Если обработка платежа воркером завершается сбоем (например, превышено число неудачных попыток отправки вебхука), сообщение отвергается (`reject` или `nack`) и автоматически перенаправляется RabbitMQ в очередь недоставленных сообщений `payments.new.dlq`.
  - В приложении зарегистрирован специальный консьюмер на DLQ, который логирует инцидент для последующего разбора администратором (DLQ Alerting).

## 🚀 Быстрый запуск

### Требования
- Установленный Docker и Docker Compose.
- Установленный Poetry (для локального запуска тестов).

### 1. Клонирование репозитория и сборка окружения
Запустите все сервисы (PostgreSQL, RabbitMQ, FastAPI API, FastStream Consumer) одной командой:
```bash
docker compose up --build -d
```
Сборка выполнит multi-stage Docker build, установит необходимые зависимости внутри контейнеров с использованием Poetry и запустит проверку готовности (healthchecks).

### 2. Применение миграций БД
Миграции Alembic применяются автоматически при старте контейнера `api`. Если вам нужно накатить их вручную, выполните:
```bash
docker compose exec api alembic upgrade head
```

### 3. Линтинг и форматирование кода
Для проверки качества кода и его форматирования используется инструмент `ruff`.
Проверить и отформатировать код локально можно с помощью команд:
```bash
poetry run ruff format app tests
poetry run ruff check app tests --fix
```

## 🧪 Запуск интеграционных тестов

Интеграционные тесты проверяют:
1. Авторизацию запросов по API-ключу.
2. Логику создания платежей и идемпотентность.
3. Полный асинхронный цикл обработки транзакции и отправки вебхуков.
4. Корректную пересылку сбойных сообщений в DLQ (Dead Letter Queue) после неудачных ретраев.

Для запуска тестов локально выполните:
```bash
poetry run pytest
```
*(Перед запуском убедитесь, что контейнеры запущены, так как тесты обращаются к реальному API и RabbitMQ на портах `8000` и `15672`).*

## 📡 Примеры API запросов (curl)

Все эндпоинты защищены API-ключом, передаваемым в заголовке `X-API-Key`. По умолчанию в конфигурации задан ключ `secret_api_key_123`.

### 1. Создание платежа
Запрос отправляет транзакцию в асинхронную обработку.

```bash
curl -X POST http://localhost:8000/api/v1/payments \
  -H "X-API-Key: secret_api_key_123" \
  -H "Idempotency-Key: a3d3d63b-6e27-4638-b715-e2db3db784d1" \
  -H "Content-Type: application/json" \
  -d '{
    "amount": 2500.50,
    "currency": "RUB",
    "description": "Оплата заказа #10492",
    "metadata": {
      "user_id": 4122,
      "item_id": 992
    },
    "webhook_url": "http://localhost:8000/api/v1/webhook-receiver"
  }'
```

**Ответ (202 Accepted):**
```json
{
  "payment_id": "8422501a-8bb7-4aeb-8f78-2d8869a8b13c",
  "status": "pending",
  "created_at": "2026-06-19T16:00:00Z"
}
```

### 2. Получение статуса платежа
Запрос позволяет проверить текущее состояние платежа.

```bash
curl -X GET http://localhost:8000/api/v1/payments/8422501a-8bb7-4aeb-8f78-2d8869a8b13c \
  -H "X-API-Key: secret_api_key_123"
```

**Ответ (200 OK):**
```json
{
  "payment_id": "8422501a-8bb7-4aeb-8f78-2d8869a8b13c",
  "amount": 2500.50,
  "currency": "RUB",
  "description": "Оплата заказа #10492",
  "metadata": {
    "user_id": 4122,
    "item_id": 992
  },
  "status": "succeeded",
  "webhook_url": "http://localhost:8000/api/v1/webhook-receiver",
  "created_at": "2026-06-19T16:00:00Z",
  "processed_at": "2026-06-19T16:00:04Z"
}
```

### 3. Тестовый приемник вебхуков (Webhook Receiver)
В сервис встроен эндпоинт для упрощения тестирования вебхуков локально. При получении уведомлений он пишет детали в лог API-контейнера:
```bash
curl -X POST http://localhost:8000/api/v1/webhook-receiver \
  -H "Content-Type: application/json" \
  -d '{
    "payment_id": "8422501a-8bb7-4aeb-8f78-2d8869a8b13c",
    "status": "succeeded",
    "amount": 2500.50,
    "currency": "RUB"
  }'
```
